🔴 问题：单例模式的隐患

单例模式常见问题
🔗 循环依赖风险
EditorService → CommandService → EditorService
难以察觉，调试困难
🧪 测试困难
全局状态污染测试环境
每个测试需要清理/重置
📦 隐式依赖
调用方不知道依赖的存在
代码难以理解
🔄 生命周期模糊
何时创建？何时销毁？
多编辑器实例时问题放大

📊 方案对比

方案 优点 缺点 适用场景
全局单例 简单直接，调用方便 循环依赖、测试困难、状态污染 无状态工具类
React Context 显式依赖，生命周期清晰 只能在组件内使用，性能开销 UI 状态共享
依赖注入容器 解耦、可测试、可替换 需要额外基础设施 复杂服务依赖
Zustand Store 项目已有、响应式、可测试 学习曲线 编辑器状态管理

✅ 推荐架构：分层管理

📦 无状态服务层 (Utility Services) → 单例
特点：无状态、纯函数、可复用
BlockRegistry CommandRegistry NodeViewRegistry
✅ 适合单例：无状态、全局共享
🔄 状态管理 (State Stores) → Zustand
特点：响应式、组件订阅、可测试
EditorStore DocumentStore SelectionStore
✅ 使用 Zustand：项目已有、支持多实例
⚙️ 业务服务 (Business Services) → 工厂创建
特点：有状态、生命周期、依赖注入
EditorService AutoSaveService AIContextService
✅ 工厂模式：支持多编辑器实例、生命周期清晰

🔧 依赖注入容器示例

// container/editor-container.ts
class EditorContainer {
  private services: Map<string, any> = new Map();

  // 注册服务（单例）
  register<T>(key: string, factory: () => T): void {
    if (!this.services.has(key)) {
      this.services.set(key, factory());
    }
  }

  // 获取服务
  get<T>(key: string): T {
    const service = this.services.get(key);
    if (!service) {
      throw new Error(`Service ${key} not found`);
    }
    return service;
  }

  // 创建编辑器实例
  createEditorInstance(documentId: string): EditorService {
    const store = createEditorStore();        // 新 Zustand store
    const autosave = new AutoSaveService(store);
    const aiContext = new AIContextService(store);

    return new EditorService(store, autosave, aiContext);
  }

  // 清理实例
  disposeEditorInstance(documentId: string): void {
    // 清理对应服务
  }
}

// 全局容器实例
export const editorContainer = new EditorContainer();

// 初始化时注册无状态服务
editorContainer.register('blockRegistry', () => new BlockRegistry());
editorContainer.register('commandRegistry', () => new CommandRegistry());

✅ 设计原则总结

  1. 无状态服务 → 单例（Registry 类）
  2. 状态管理 → Zustand Store（响应式、可测试）
  3. 业务服务 → 工厂创建（支持多实例、生命周期清晰）
  4. 依赖注入容器 管理服务创建和获取
  5. 显式依赖 通过构造函数注入，避免循环依赖
