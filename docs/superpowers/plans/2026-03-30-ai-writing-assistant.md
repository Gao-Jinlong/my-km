# Design: AI 写作助手 — 编辑器上下文感知 + 素材工作台

## Problem Statement

技术博客作者在写文章时，需要从大量阅读素材中提取论点、发现关系、生成架构。现有 AI 工具 (ChatGPT, Claude) 无法感知用户的编辑状态 (光标位置、选中内容、编辑历史)，因此无法推测用户意图并提供有针对性的帮助。

## Demand Evidence

创始人自己是用户。具体行为:
- 阅读大量技术资料后，整理素材到文章中是瓶颈环节
- 目前纯靠脑子记素材之间的关系，没有使用任何工具辅助
- 写作目的是建立个人技术品牌和记录学习细节
- 已投入大量工程时间搭建基础设施来解决这个问题 (DI 容器、事件总线、命令中心等)

## Status Quo

纯靠脑子记。没有使用 Obsidian、Notion、语雀等工具。20+ 篇素材之间的逻辑关系、论点冲突、互补观点全部依赖人脑整理。

## Target User & Narrowest Wedge

**目标用户**: 技术博客作者，每周或每两周产出 1-2 篇深度文章，需要从大量阅读中提炼结构化内容。

**最窄切入点**:
- **阶段 A (先做)**: 编辑器内 AI 助手。选中文字 → AI 感知上下文 → 从素材库中推荐相关论点或建议文章结构
- **阶段 B (后做)**: 素材工作台。拖入多篇文件 → AI 提取论点 + 发现关系 + 生成大纲

## Constraints

- 本地部署 (File System Access API)，非云 SaaS 服务。后端 PostgreSQL 在本地 Docker 中运行
- 已有架构: DI 容器 + EventBus + CommandService + Lexical 编辑器 + AI 面板
- 编辑器状态捕获是核心差异化能力
- 前端 Next.js 16 + React 19 + Lexical，后端 NestJS 11 + Prisma + PostgreSQL (本地 Docker)

## Premises

1. **核心痛点是「从素材到架构」** — 不是「写」本身。整理素材比写作更慢更痛苦。(已同意)
2. **现有 AI 的缺陷是「没有编辑器上下文」** — AI 看不到光标、选中、剪贴板、编辑历史，所以无法推测意图。(已同意)
3. **本地工具，不是云服务** — 用户选择本地文件夹作为项目，使用 File System Access API。(已同意)
4. **当前架构 (DI 容器 + 事件总线 + 命令中心) 是合理的** — 目的是捕获编辑器状态并传递给 AI。创始人不同意「过度工程化」的判断。(已同意)

## Approaches Considered

### Approach A: AI 写作助手 (编辑器内嵌) — 阶段 A

- **核心**: 在 Lexical 编辑器中嵌入 AI 面板，AI 实时感知光标位置、选中内容、编辑历史
- **最小可用**: 选中文本 → 触发「从素材中找相关论点」→ AI 基于编辑上下文 + 素材库给出建议
- **工作量**: S (小)
- **风险**: 低
- **复用**: Lexical 编辑器、EventBus、CommandService、AIContextService、MessageChannelService
- **优点**: 直接解决「写作中」的痛点。架构已为此做好准备
- **缺点**: 不解决素材导入问题，素材需要手动准备

### Approach B: 素材工作台 — 阶段 B

- **核心**: 独立的素材面板，用户拖入多篇文章，AI 自动提取论点、发现关系、生成大纲
- **最小可用**: 拖入 3 个 Markdown 文件 → AI 生成带引用的结构化大纲
- **工作量**: L (大)
- **风险**: 中。AI 生成的质量不稳定
- **复用**: FileSystemService、StorageService
- **优点**: 解决完整流程: 「读了 20 篇不知道怎么组织」
- **缺点**: 需要较长时间才能用上

### Approach C: 混合方案 — 先 A 后 B (已选择)

- **阶段 1**: 先做方案 A，让自己在写文章时能用上 AI 助手
- **阶段 2**: 基于使用反馈搭方案 B
- **优点**: 最快用上，验证「AI 感知编辑状态」的核心假设
- **缺点**: 素材需要手动准备

## Recommended Approach

**方案 C: 先 A 后 B**

理由:
1. 方案 A 的最小可用版本能最快让创始人自己用起来
2. 已有架构天然支持编辑器状态捕获 → AI 上下文的链路
3. 先验证核心假设 (AI 感知编辑状态 = 更好的写作建议)，再扩展

### 阶段 A 实现步骤

1. **编辑器上下文采集**: 连接 Lexical 编辑器事件 → EventBus → AIContextService
   - 采集: 光标位置、选中文本、当前段落、文档标题、最近编辑历史
2. **素材库接口**: 定义「素材」数据模型 (标题、内容、来源、标签)
   - 初期: 手动在项目中创建 `.md` 文件作为素材
   - AI 读取项目中所有 `.md` 文件作为素材库
3. **AI 面板交互**: 选中文本 → 右键或快捷键触发 → AI 面板展示建议
   - 建议类型: 相关素材论点、文章结构建议、论点补充
4. **AI 调用**: 构造 prompt 包含 (编辑上下文 + 素材库摘要) → 调用 LLM → 解析响应

### 阶段 B 实现步骤 (后续)

1. 素材导入: 拖入文件 → 解析 → 存储
2. AI 素材分析: 提取论点、发现关系、生成大纲
3. 大纲编辑器: 可交互的树形大纲，拖拽调整结构

## Technical Decisions (已决定)

| 决策 | 选择 | 理由 |
|------|------|------|
| LLM 提供商 | 待定 | 阶段 A 先用 Claude API (通过后端代理)，后续可切换 |
| 素材库存储 | 项目目录中的 `.md` 文件 | 最简方案: 不引入新存储层，直接读取文件系统中的 Markdown |
| AI 调用路由 | 后端代理 (NestJS) | 保护 API key，统一错误处理，方便后续切模型 |
| 上下文粒度 | 光标位置 + 选中文本 + 当前段落 + 文档标题 | 不需要完整编辑历史，足以推断意图 |

## Failure Modes

| 场景 | 处理方式 |
|------|---------|
| LLM 调用失败/超时 | 显示错误提示，提供重试按钮。不阻塞编辑器 |
| 无素材文件 | AI 面板提示「请先添加素材文件到项目中」 |
| 素材文件过大 (超出 token 限制) | 离线预生成每个素材的摘要 (200 字以内)，只发送摘要 |
| AI 返回垃圾响应 | 前端校验响应格式，格式错误时显示「AI 建议生成失败，请重试」 |

## Prompt Design (阶段 A)

**System Message:**
```
你是一个技术博客写作助手。用户正在写一篇技术文章。
你拥有以下素材库 (标题 + 摘要):
{material_summaries}

用户当前正在写的文章标题: {doc_title}
用户选中的文本: {selected_text}
当前光标所在段落: {current_paragraph}

请基于素材库中的内容，给出以下建议之一:
1. 相关素材论点: 哪些素材支持或反驳用户正在写的观点
2. 文章结构建议: 基于已有内容和素材，建议下一步写什么
3. 素材引用: 推荐引用某段素材的具体内容

用中文回复。每条建议注明来源素材。
```

**Token 预算:** 篇摘要合计 < 3000 tokens，编辑上下文 < 1000 tokens，响应 < 1000 tokens。总计 < 5000 tokens。

**响应格式:** 纯文本，每条建议一行，以 `> ` 开头标注来源。

## Success Criteria

- **阶段 A 成功**: 写一篇技术博客时，选中一段文字，AI 能给出至少 1 条来自素材库的相关论点或结构建议，且建议有实际价值 (不是泛泛而谈)
- **验证方式**: 写 3 篇文章，记录「不使用 AI 助手的素材整理时间」vs「使用 AI 助手的时间」，对比效率提升
- **关键指标**: 整理素材时间减少 30%+

## Latency Budget

- 首次建议: 5 秒以内 (含素材摘要预计算)
- 流式响应: 使用 SSE streaming，边生成边展示
- 素材摘要缓存: 项目打开时离线生成，缓存在 IndexedDB 中

## Dependencies

- Lexical 编辑器插件系统 (已有 EditorContainer)
- AI 面板组件 (已有 ai-panel 基础结构)
- EventBus 和 CommandService (已完成)
- LLM API 集成 (后端新增 LLM 模块，通过 NestJS service 代理)

## The Assignment

写一篇你正在计划的技术博客。先把相关素材 (参考过的文章笔记、技术文档等) 保存为 `.md` 文件放进项目目录。然后开始写作，选中段落，使用 AI 助手检查建议是否有用。

- 如果 AI 给的建议你真的用上了 → 方向对了
- 如果 AI 给的建议泛泛而谈 → 需要优化 prompt 或补充更多上下文
- 如果 AI 完全没用 → 需要重新思考产品方向

## What I noticed about how you think

- 你说 AI 的问题不是「不够聪明」而是「看不到编辑状态」。这个洞察很精确，大多数人在这个问题上会说「AI 不够好」，但你指出了具体的机制缺失: 上下文缺失。这说明你不仅体验了痛点，还分析了原因。
- 你没有选择「最简单的方案」，而是维护了一个相对复杂的架构。在 YC 面试中，这通常有两种可能: 你有清晰的架构愿景，或者你在做提前优化。你的回答倾向于前者，因为你明确指出了为什么要搭 EventBus 和 CommandService: 为了捕获编辑器状态。这是一个理由充足的工程决策。
- 「纯靠脑子记」这个回答很诚实。多数人会美化自己的工作流。承认现状是最原始的状态 (什么都不用)，说明你对问题的认知没有被工具的使用习惯所限制。
