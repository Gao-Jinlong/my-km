# Trace Context Ownership Design

> **Date:** 2026-06-14
> **Status:** Approved
> **Scope:** Frontend tracing architecture refactor

## Problem

Current frontend tracing uses `TracingService.activeTraceparent` — a singleton global mutable field — to propagate trace context to HTTP requests and tool execution. This breaks under concurrent conversations: if conversation A sets the traceparent and conversation B overwrites it before A's request fires, A's trace chain is silently severed.

Additionally, each `sendMessage` call generates a brand-new `traceId`, so messages within the same conversation appear as disconnected traces rather than a unified call chain.

## Goals

1. **Eliminate global mutable trace state** — `TracingService` becomes stateless per DDD application-layer principles.
2. **Thread-level trace correlation** — all messages within a conversation share one `traceId`.
3. **Explicit context passing** — trace context flows through the call chain as a value object, not through global state.
4. **Architecture-ready for concurrent conversations** — even though multi-panel UI is not yet implemented, the design must not block it.

## Architecture

### DDD Layering

| Layer | Responsibility | Components |
|-------|---------------|------------|
| Domain | Trace data models (pure, no dependencies) | `ActiveSpan`, `SpanData`, `SpanEvent`, `TraceContext` |
| Application | Stateless span orchestration | `TracingService` — `startSpan`, `endSpan`, `getTraceparent`, `toTraceContext` |
| Infrastructure | Span export | `BrowserSpanExporter` |
| Consumer | Trace context ownership & propagation | `useLangGraphStream`, `AIPanel`, `FrontendToolExecutor` |

### Trace Context Flow

```
useLangGraphStream (owns trace context via refs)
  │
  ├── sendMessage()
  │     ├── get/create thread-level traceId (Map<threadId, traceId>)
  │     ├── create root span (reuse traceId)
  │     ├── set traceparentRef (for onRequest closure)
  │     └── submit(values, { metadata: { __trace: { traceId } } })
  │           └── per-instance client onRequest → withTraceparent middleware
  │                 └── HTTP header: traceparent: 00-{traceId}-{spanId}-01
  │
  ├── return { traceContext }  (exposed to AIPanel)
  │
  └── AIPanel
        └── toolExecutor.dispatch(name, input, { traceContext })
              └── FrontendToolExecutor creates child span with traceContext
```

### Span Hierarchy (Thread-Level Trace)

```
Trace T1 (thread-level, generated by frontend on first message)
├── frontend.chat.sendMessage (message 1, spanId=S1)
│   └── backend: langgraph.run
│       └── backend: llm_node.invoke
├── frontend.tool.execute (tool call in message 1)
└── frontend.chat.sendMessage (message 2, spanId=S2)
    └── backend: langgraph.run
        └── backend: llm_node.invoke
```

Each message is a sibling span under the same trace. Backend spans auto-link via W3C `traceparent` header.

## Design Details

### 1. TracingService — Stateless

**Remove:**
- `activeTraceparent` field
- `setActiveTraceparent()` / `getActiveTraceparent()` methods
- Corresponding entries in `ITracingService` interface

**Keep:**
- `startSpan(name, options)` — creates `ActiveSpan`
- `endSpan(span)` — ends span and exports via `BrowserSpanExporter`
- `getTraceparent(traceId, spanId)` — formats W3C traceparent string
- `forceFlush()` — flush exporter
- `dispose()` — cleanup

**Add:**
- `toTraceContext(span: ActiveSpan): TraceContext` — extracts immutable context snapshot from a span

### 2. TraceContext — Value Object

```typescript
/** Immutable trace context snapshot for explicit passing through call chains */
export interface TraceContext {
    readonly traceId: string;
    readonly spanId: string;
    readonly traceparent: string;
}
```

Created once from an `ActiveSpan`, then passed explicitly. No global state involved.

### 3. useLangGraphStream — Trace Context Owner

The hook becomes the sole owner of trace context for its conversation lifecycle.

**Refs:**

```typescript
// threadId → traceId mapping, supports future multi-conversation switching
const threadTraceIds = useRef<Map<string, string>>(new Map());

// temporary traceId for first message (threadId is null until backend creates the thread)
const pendingTraceId = useRef<string | null>(null);

// current request span and traceparent snapshot
const activeTraceSpan = useRef<ActiveSpan | null>(null);
const traceparentRef = useRef<string | null>(null);
```

> **pendingTraceId lifecycle:** On the first message, `threadId` is `null` because the thread is created by the backend during `stream.submit()`. The traceId is stored in `pendingTraceId` temporarily. When `onThreadId` fires (backend returns the new thread ID), the pending traceId is persisted to `threadTraceIds` map. Subsequent messages find their traceId in the map directly.

**sendMessage flow:**

```typescript
// 1. Look up existing thread-level traceId (if any)
const existingTraceId = threadId ? threadTraceIds.current.get(threadId) : undefined;

// 2. Create root span — reuse traceId if exists, auto-generate if not
const rootSpan = tracer.startSpan('frontend.chat.sendMessage', {
    ...(existingTraceId ? { traceId: existingTraceId } : {}),
    attributes: { 'chat.messageLength': content.length },
});

// 3. Persist traceId — store in map if threadId known, or pendingTraceId if thread not yet created
if (threadId) {
    if (!threadTraceIds.current.has(threadId)) {
        threadTraceIds.current.set(threadId, rootSpan.traceId);
    }
} else {
    pendingTraceId.current = rootSpan.traceId;
}

// 4. Set traceparentRef for onRequest closure
traceparentRef.current = tracer.getTraceparent(rootSpan.traceId, rootSpan.spanId);
activeTraceSpan.current = rootSpan;

// 5. Submit with metadata for backend thread-level correlation
await stream.submit(
    { messages: [...] },
    {
        context,
        metadata: { __trace: { traceId: rootSpan.traceId } },
    },
);
```

**onThreadId callback (persist pending traceId):**

```typescript
onThreadId: id => {
    if (pendingTraceId.current && !threadTraceIds.current.has(id)) {
        threadTraceIds.current.set(id, pendingTraceId.current);
        pendingTraceId.current = null;
    }
    activeTraceSpan.current?.addEvent('metadata_received', { threadId: id });
    setThreadId(id);
},
```

**resumeWithToolResult flow:**

```typescript
const resumeSpan = tracer.startSpan('POST /runs/resume', {
    traceId: threadTraceIds.current.get(threadId ?? '') ?? activeTraceSpan.current?.traceId,
    parentSpanId: activeTraceSpan.current?.spanId,
    ...
});
traceparentRef.current = tracer.getTraceparent(resumeSpan.traceId, resumeSpan.spanId);
```

**Stream end effect:**

```typescript
// Clear per-request state (traceId persists in threadTraceIds map)
traceparentRef.current = null;
activeTraceSpan.current = null;
```

**Return value:**

```typescript
return {
    ...,
    traceContext: activeTraceSpan.current
        ? tracer.toTraceContext(activeTraceSpan.current)
        : null,
};
```

### 4. langgraph-client.ts — Factory + Middleware

**Delete:**
- `createLangGraphRequestHook()` — read global state, no longer needed

**Add `withTraceparent` middleware:**

```typescript
/**
 * Traceparent injection middleware.
 * Reads current traceparent via getter, injects into request header.
 */
export function withTraceparent(
    getTraceparent: () => string | null,
): (url: URL, init: RequestInit) => RequestInit {
    return (_url, init) => {
        const tp = getTraceparent();
        if (!tp) return init;
        const headers = new Headers(init.headers);
        headers.set('traceparent', tp);
        return { ...init, headers };
    };
}
```

**Refactor `createClient` → `createLangGraphClient`:**

```typescript
export function createLangGraphClient(options?: {
    onRequest?: (url: URL, init: RequestInit) => RequestInit;
}): Client {
    return new Client({
        apiUrl: API_URL,
        ...(options?.onRequest ? { onRequest: options.onRequest } : {}),
    });
}
```

**Bare singleton for thread CRUD (no tracing):**

```typescript
export const langgraphClient = createLangGraphClient();
```

**Consumer usage in useLangGraphStream:**

```typescript
const client = useMemo(
    () => createLangGraphClient({
        onRequest: withTraceparent(() => traceparentRef.current),
    }),
    [],
);
```

### 5. FrontendToolExecutor — Explicit Context

**ToolDispatchOptions:**

```typescript
export interface ToolDispatchOptions {
    toolCallId?: string;
    traceContext?: TraceContext; // new
}
```

**ToolTracingService (simplified):**

```typescript
export interface ToolTracingService
    extends Pick<ITracingService, 'startSpan' | 'endSpan'> {} // removed getActiveTraceparent
```

**dispatch() uses passed context:**

```typescript
const spanOptions: SpanOptions = {
    traceId: options?.traceContext?.traceId,
    parentSpanId: options?.traceContext?.spanId,
    attributes: { 'tool.name': toolName, ... },
};
const toolSpan = tracer.startSpan('frontend.tool.execute', spanOptions);
```

**Delete `parseTraceparent()`** — no longer parsing traceparent strings; receive structured `TraceContext` directly.

### 6. AIPanel — Pass-Through

```typescript
const { ..., traceContext } = useLangGraphStream();

useEffect(() => {
    if (!interrupt) return;
    toolExecutor.dispatch(interrupt.toolName, interrupt.input, {
        toolCallId: interrupt.toolCallId,
        traceContext,
    }).then(result => {
        resumeWithToolResult(interrupt.toolCallId, result);
    });
}, [interrupt, toolExecutor, resumeWithToolResult, traceContext]);
```

## Consumer Client Matrix

| Consumer | Purpose | Client |
|----------|---------|--------|
| `useLangGraphStream` | run streaming (needs traceparent) | `createLangGraphClient({ onRequest })` per-instance |
| `ConversationList` etc. | thread CRUD (no traceparent) | `langgraphClient` bare singleton |

## Backend Impact

**No backend changes required.** Backend already extracts trace context from `traceparent` HTTP header via standard OTel Node SDK instrumentation. The `metadata.__trace.traceId` is a frontend-owned supplementary correlation signal; backend may optionally store it in thread metadata in the future, but that is out of scope.
