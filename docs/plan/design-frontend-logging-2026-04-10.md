## Problem Statement

前端调试过程混乱：每次调试都需要临时添加 `console.log`，调试完成后忘记删除或不想删，导致代码中堆积了约 48 处裸 `console.*` 调用（platform 服务 37 处 + dialog UI 9 处 + 零星 2 处，不含测试和 ConsoleWriter 自身）。最近的两次 `debug:` 提交（e11f7b4, 35ed121）就是典型例子。

## What Makes This Cool

想象你在调试 EventBus 和 DialogService 的交互问题。你不再需要加 `console.log`。你打开日志面板，选中 `event-bus` 和 `dialog` 两个分类，级别设为 DEBUG，立刻看到它们之间的完整交互时序。调完关掉面板，不用删任何代码。

## Constraints

- 不引入外部日志库（winston、pino 等），激活现有 `LoggerService`
- 日志存储使用 IndexedDB（持久化，跨页面刷新保留）+ 内存 ring buffer（快速读取）
- 前后端 trace ID 打通不在本次范围内（后续任务）
- 日志面板仅在开发环境加载，零生产成本

## Premises

1. **激活现有 LoggerService，不造新轮子**。`LoggerService` 已有级别、分类、LogWriter 抽象，只需注册到 DI 容器并让服务使用它。
2. **IndexedDB + 内存 ring buffer 双层存储**。IndexedDB 持久化跨刷新，ring buffer 提供面板快速读取。
3. **前后端 trace ID 打通不在本次范围内**。先把前端日志体系理顺。
4. **ConsoleWriter 的 ANSI 转义码需要修复**。当前用 `\x1b[36m` 等终端颜色码，在浏览器 DevTools 中会显示乱码。

## Approaches Considered

### Approach A: 最小可用（仅服务层）
- Summary: 把 LoggerService 注册到 DI，替换 console.*，不加 UI
- Effort: S
- Risk: Low
- Completeness: 4/10
- Pros: 最快能用，结构化日志立即可用
- Cons: 查看日志仍靠 DevTools，调试体验提升有限

### Approach B: 完整方案（服务层 + 浏览器日志面板）✅ SELECTED
- Summary: LoggerService 接入 DI + IndexedDB 持久化 + 浏览器内日志面板
- Effort: M
- Risk: Low-Medium
- Completeness: 8/10
- Pros: 完整调试体验，按模块/级别过滤，日志持久化，零生产成本
- Cons: 多一个 UI 组件需要设计
- Reuses: 现有 LoggerService、EventBusService、DI 容器

### Approach C: EventBus 驱动的可观测层
- Summary: Logger + EventBus 深度集成，带录制+导出功能
- Effort: L
- Risk: Medium
- Completeness: 10/10
- Pros: 所有事件流可观测，录制+导出对 AI 辅助调试有价值
- Cons: 改动范围大，耦合增加

## Recommended Approach

**方案 B: 完整方案**。在"够用"和"好玩"之间的甜蜜点。

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                      各平台服务                               │
│  (EventBus, Dialog, Command, Editor, Panel, ...)            │
│  每个服务通过 @Inject() 注入 LoggerService                    │
│  使用 this.logger.debug/info/warn/error() 记录日志            │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                    LoggerService                             │
│  ┌─────────────────┐  ┌──────────────────┐                  │
│  │  ConsoleWriter    │  │ IndexedDBWriter    │                  │
│  │  (修复后, %c样式) │  │ (持久化, 批量写入) │                  │
│  └─────────────────┘  └──────────────────┘                  │
│                                                              │
│  getLogger(category) → Logger 实例                           │
│  history[] → 内置 ring buffer (内存, 2000 条, 快速读取)       │
│  getHistory() → 面板读取接口                                  │
│  _onLogChange → Emitter (防抖 100ms, 面板订阅)               │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                  LogPanel (开发环境组件)                       │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ 过滤栏: [级别 ▼] [分类 ▼] [搜索...] [暂停] [清空]       │ │
│  ├────────────────────────────────────────────────────────┤ │
│  │ 12:34:56.789 [DEBUG] [event-bus] 发布事件: dialog.open │ │
│  │ 12:34:56.790 [INFO]  [dialog]    打开确认对话框          │ │
│  │ 12:34:56.791 [WARN]  [panel]     面板不存在: sidebar    │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  通过快捷键或浮动按钮触发 (如 Ctrl+Shift+L)                    │
│  数据源: LoggerService.getHistory() + 实时推送                │
└─────────────────────────────────────────────────────────────┘
```

### 实现步骤

#### Step 0: 给 LoggerService 添加 @Service() 装饰器（~2 min）

**文件**: `apps/web/src/platform/logger/service.ts`

当前 `LoggerService` 没有 `@Service()` 装饰器，DI 容器无法正确注册它。

```typescript
import { Service } from '@/platform/di';

@Service({ singleton: true })
export class LoggerService extends ServiceBase implements ILoggerService {
    // ... 现有代码不变
}
```

#### Step 1: 修复 ConsoleWriter（~5 min）

**文件**: `apps/web/src/platform/logger/writers/console.ts`

问题：当前使用 ANSI 转义码 (`\x1b[36m`)，在浏览器 DevTools 中显示乱码。

修复：使用浏览器 DevTools 原生的 `%c` 格式化，或直接使用 `console.*` 方法的内置样式。

```typescript
// 修复后
write(entry: LogEntry): void {
    const time = new Date(entry.timestamp).toISOString().slice(11, 23); // HH:mm:ss.SSS
    const levelStr = LogLevelToString(entry.level);
    const prefix = `[${time}] [${levelStr}] [${entry.category}]`;

    const style = `color: ${LEVEL_COLORS[entry.level]}; font-weight: bold;`;

    switch (entry.level) {
        case LogLevel.DEBUG:
            console.debug(`%c${prefix}`, style, entry.message, ...(entry.data || []));
            break;
        // ... 其他级别类似
    }
}
```

#### Step 2: 添加 IndexedDBWriter（~15 min）

**新文件**: `apps/web/src/platform/logger/writers/indexeddb.ts`

实现 `LogWriter` 接口：
- 打开 IndexedDB database `my-km-logs`
- Object store: `logs`，索引: `timestamp`, `category`, `level`
- 写入时 batch（每 100ms 或 50 条 flush 一次）
- 保留最近 10000 条，自动清理旧数据
- 提供 `query(filter)` 方法供面板读取

#### Step 3: 注册 LoggerService 到 DI 容器（~5 min）

**文件**: `apps/web/src/platform/bootstrap.ts`

```typescript
// 1. 导入 LoggerService
import { LoggerService } from './logger/service';

// 2. 在 AppServices 接口添加
loggerService: LoggerService;

// 3. 在 createServiceContainer 中注册（必须在其他服务之前）
container.register(LoggerService);

// 4. 初始化时添加 IndexedDBWriter
const loggerService = container.get(LoggerService);
loggerService.addWriter(new IndexedDBWriter());
```

关键：LoggerService 必须最先注册，因为其他服务依赖它。

#### Step 4: 迁移所有服务（~20 min）

逐个服务迁移，使用与现有 DI 一致的模式：

```typescript
// 迁移前
console.warn(`[PanelService] 面板不存在: ${panelId}`);

// 迁移后 — 使用 @Inject() 构造函数注入
import { Inject, Service } from '@/platform/di';
import { LoggerService } from '@/platform/logger';
import type { Logger } from '@/platform/logger';

@Service({ singleton: true })
export class PanelService extends ServiceBase {
    private readonly logger: Logger;

    constructor(
        @Inject(LoggerService) private loggerService: LoggerService,
        // ... 其他依赖
    ) {
        super();
        this.logger = loggerService.getLogger('panel');
    }

    // 使用处
    this.logger.warn('面板不存在: {id}', panelId);
}
```

**注意**: 注入 `LoggerService` 会改变构造函数参数列表，需要检查 `@Inject()` 位置索引是否正确。`MessageChannelService` 有非标准构造函数 `(config, options)`，需要特殊处理。

**React hooks/组件中的日志**（如 `use-dialogs.ts`、`dialog-provider.tsx`）不能使用构造函数注入，需要直接从容器获取：
```typescript
const loggerService = container.get(LoggerService);
const logger = loggerService.getLogger('dialog');
```

需要迁移的文件（按 console.* 数量排序）：
1. `platform/panel/service.ts` — 12 处
2. `platform/message-channel/service.ts` — 7 处
3. `platform/file-open/service.ts` — 6 处
4. `platform/command/service.ts` — 4 处
5. `platform/conditional/service.ts` — 3 处
6. `platform/context-menu/service.ts` — 2 处
7. `platform/event-bus/service.ts` — 2 处
8. `platform/di/container.ts` — 1 处
9. `platform/keyboard/shortcut.service.ts` — 1 处
10. `components/dialog/use-dialogs.ts` — 6 处（临时调试日志，直接删除）
11. `components/dialog/dialog-provider.tsx` — 3 处（临时调试日志，直接删除）

#### Step 5: 构建日志面板组件（~30 min）

**新文件**: `apps/web/src/components/debug/log-panel.tsx`

功能清单：
- **过滤栏**: 级别下拉、分类多选、搜索框、暂停/继续按钮
- **日志列表**: 简单滚动容器（`overflow-y: auto`），按时间倒序显示。10000 条以内的日志不需要虚拟滚动
- **颜色编码**: DEBUG=蓝, INFO=绿, WARN=黄, ERROR=红
- **展开详情**: 点击日志条目展开查看 `data` 字段
- **快捷键**: `Ctrl+Shift+L` 打开/关闭面板
- **位置**: 底部浮动面板，可拖拽高度
- **环境守卫**: 仅在 `process.env.NODE_ENV === 'development'` 时渲染

数据流：
```
LoggerService → history[] (内置 ring buffer)
                    ↓
LogPanel 组件通过 container.get(LoggerService).getHistory() 读取
实时更新: LoggerService 暴露 _onLogChange Emitter (防抖 100ms)，面板订阅
注意: onChange 必须防抖，避免高频日志（如 EventBus 事件风暴）导致 React 过度渲染
```

## Open Questions

1. 日志面板是否需要支持导出日志为 JSON 文件？（当前设计不含，可以作为后续增强）
2. IndexedDB 的存储上限策略：按条数（10000 条）还是按时间（保留 7 天）？

## What I noticed about how you think

- 你说"调试过程混乱"而不是"需要更好的日志"，这说明你在意的是**工作流**而不是技术实现。这个直觉是对的，工具应该适应人的习惯，而不是反过来。
- 你选择了"先服务层，后 UI"而不是"一起来"，这表明你喜欢分步验证、确认每一步能用再继续。这是好的工程节奏。
- 你坚持要 IndexedDB 而不是简单的 ring buffer，因为你想到"刷新页面日志就没了"这个场景。这种从实际使用场景反推需求的能力很有价值。
