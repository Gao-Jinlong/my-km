## Problem Statement

当前 `AiGateway` (`apps/server/src/ai/gateway/ai-ws.gateway.ts`) 存在三个核心问题：

1. **网关耦合业务逻辑** — 直接调用 `ConversationService.create()`, `MessageService.findByConversationId()` 等持久层服务，违背了网关只负责消息路由的职责
2. **Socket.io 房间概念误用** — 使用 `client.join(roomId)` 和 `server.to(roomId).emit()` 做消息分发，这是为"多个物理连接共享一个对话"设计的，但前端是单个 Socket 连接（`WSClientService` 全局单例），需要的是"单个连接内按逻辑 room 隔离状态"
3. **模块边界不清** — WebSocket 网关放在 `ai/` 模块下，与 AI 业务代码强耦合

## Naming: conversation → room

当前用 `conversation` 表示一次 AI 对话。后续会实现多个 AI 在同一个 room 中对话的能力，`conversation` 语义过窄。统一用 `room` 替换：
- 服务端：`ConversationService` → `RoomService`（Prisma model 也重命名）
- 网关事件：`conversationId` → `roomId`
- 状态机：按 roomId 创建实例

## Message Flow (after refactor)

```
Client (single WS conn, /ai namespace)
  │
  │  emit({ type: "create_and_send", roomId: "...", content: "..." })
  │  emit({ type: "send_message", roomId: "...", content: "..." })
  │  emit({ type: "join", roomId: "..." })
  │  emit({ type: "stop", roomId: "..." })
  │  emit({ type: "tool_result", roomId: "...", toolCallId: "...", result: ... })
  ▼
WsGateway (ws/ module, thin router)
  │
  ├── route by message type
  │
  ▼
RoomRouter (ai/ module)
  │
  ├── create_and_send → RoomService.create() + RoomStateMachineFactory.create()
  │                     + RequestDispatcher.dispatch()
  │
  ├── send_message    → RoomService.findById() + RoomStateMachineFactory.create()
  │                     + RequestDispatcher.dispatch()
  │
  ├── join            → RoomService.findById() + MessageService.findByRoomId()
  │                     + emit history
  │
  ├── stop            → RoomStateMachineFactory.get(roomId).stop()
  │
  └── tool_result     → ToolDispatcher.deliverResult()
                        (or delegate to WorkflowExecutor)

RoomStateMachineFactory
  │
  ├── create(roomId)  → new RoomStateMachine(roomId) + register
  ├── get(roomId)     → existing FSM
  ├── destroy(roomId) → deregister + cleanup
  └── onDisconnect(clientId) → destroy all FSMs for this client

RoomStateMachine (per-room instance)
  │
  ├── States: Idle → BuildingContext → Processing → [ToolWaiting → ToolExecuting → Processing]* → Done
  ├── emit(event) → callback → WsGateway.emitToRoom(roomId, event)
  └── AbortController per session
```

## Key Design Decisions

### 1. WsGateway 位置
**Decision**: 将 WebSocket 网关拆为独立 `ws/` 模块（`apps/server/src/ws/`），不放在 `ai/` 下。
**Why**: WebSocket 是传输层基础设施，不应与 AI 业务耦合。后续如果有其他 WS 场景可复用。

### 2. 消息路由模式
**Decision**: WsGateway 只做类型分发，调用 `RoomRouter` 的对应方法。`RoomRouter` 在 `ai/` 模块中，负责编排各服务。
**Why**: 网关保持纯净（只有路由逻辑），业务编排集中在 Router 层，各 Service 保持单一职责。

### 3. 连接内消息分发（替代 Socket.io rooms）
**Decision**: 不再使用 `client.join(roomId)` 和 `server.to(roomId).emit()`。改为：
- `RoomStateMachineFactory` 管理所有 room 的状态机实例
- 每个 FSM 实例注册一个 `emit` 回调，该回调通过 WsGateway 的 `server.to(socketId).emit()` 发给指定客户端
- 或者更简单地：WsGateway 维护 `clientId → Socket` 映射，按 clientId 直接 emit
**Why**: 当前架构是单连接多 room，不需要 Socket.io 的多连接广播能力。直接按 clientId 路由更简单。

### 4. conversation → room 重命名范围
**Decision**: 拆到独立 PR，不混入本次架构重构。
- **本 PR**（架构重构）：保持 `conversationId` 名称不变，只重构模块和路由逻辑
- **后续 PR**（纯重命名）：传输层事件字段 `conversationId` → `roomId`，前后端 41 个文件同步改名
**Why**: 架构重构已经涉及大量逻辑变更，叠加纯机械重命名会让 code review 难以聚焦。两个 PR 独立审查，降低风险。

### 5. ConnectionManager 去留
**Decision**: 移除 `ConnectionManager`。其功能被以下组件替代：
- clientId → Socket 映射：由 WsGateway 内部 Map 维护
- 连接/断开生命周期：由 WsGateway 的 handleConnection/handleDisconnect 处理
- room 成员追踪：不再需要（单连接架构下，room 成员就是当前客户端自己）
**Why**: ConnectionManager 的设计假设是多连接多 room，与当前单连接多 room 架构不匹配。

## Approaches Considered

### Approach A: 最小修改（在当前 AiGateway 上修补）
- 只在现有 Gateway 中抽离业务逻辑到 Router，保留 Socket.io rooms
- Effort: S (1-2 天)
- Risk: Low
- Pros: 改动最小；风险可控
- Cons: 不解决 Socket.io rooms 误用问题；模块仍然耦合在 ai/ 下；治标不治本
- 不推荐

### Approach B: 完整重构（拆 ws 模块 + RoomRouter + RoomStateMachine + 单连接分发）
- Effort: M (3-5 天)
- Risk: Medium
- Pros: 职责清晰；连接资源高效；为未来多 AI room 功能打基础；协议更简洁
- Cons: 改动面广（前后端协议 breaking change）；需要完整的回归测试

### Approach C: 折中（保留 AiGateway 名称但拆 ws 到独立模块，暂不重命名 conversation）
- Effort: M (3-4 天)
- Risk: Medium
- Pros: 模块解耦到位；协议保持兼容；可以逐步重命名
- Cons: conversation 名称暂时保留，语义不准

**推荐 Approach B** — 一次性解决架构问题，前后端同步更新协议。

## Constraints

- NestJS + Socket.io 技术栈不变
- 前端 `WSClientService` 保持单连接模型
- Prisma 数据库 schema 尽量不改（避免 migration 风险）
- 现有的 LangGraph workflow runtime、ToolDispatcher、AISessionManager 保持不变

## Implementation Order

1. 创建 `apps/server/src/ws/` 模块（WsGateway + SocketRegistry 替代 ConnectionManager）
2. 创建 `RoomRouter` 类（从当前 AiGateway 迁移业务编排逻辑，明确与 RequestDispatcher 边界）
3. 重构 `RoomStateMachineFactory`（从当前 ConversationStateMachine 改造，per-room FSM 实例）
4. 统一 ConversationStateMachine.stop/error 方法调用 _getByConvOrThrow
5. 提取 Gateway 共享 emitError 方法，消除重复 try/catch
6. WorkflowExecutor 改为 EventEmitter 回调机制，解耦对 ConversationStateMachine 的直接调用
7. 将 AiController 的 POST /ai/chat 重定向到 RequestDispatcher，移除 AiService
8. 移除旧的 AiGateway 和 ConnectionManager
9. 移除 Socket.io room join/to 用法，改为按 clientId 直接 emit
10. 更新所有测试，补充 WsGateway、RoomRouter、SocketRegistry 测试用例
