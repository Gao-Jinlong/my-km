## Problem Statement

当前后端 AI 对话实现存在以下问题：
- RoomStateMachineFactory 用单例工厂管理所有 room 的 FSM 实例，导致传输层和业务层耦合
- RoomOrchestrator 直接依赖 RoomStateMachineFactory，违反依赖倒置
- 两套状态追踪系统（FSM + AISessionManager）重叠且可能不同步
- Executor 对 FSM 有直接依赖，应该通过回调接口解耦

目标：重新设计从收到消息到返回响应的完整流程，实现状态和代码完全无关，每个执行单元通过实例传递上下文。

## Premises

1. WS Service → MessageBus → AI Router 的消息分发模式保持不变
2. Room 是对话的基本单位，一个 Room 可以有多个 AI 参与
3. Executor 每次执行创建独立实例，执行完销毁（Instance-per-execution）
4. 上下文构建和压缩是独立组件，不耦合在 Executor 内部
5. 状态机实例在调用链中传递，消除全局工厂查找
6. 回调接口是 Executor 和 transport 层的唯一连接点

## Data Flow (create_and_send)

```
Client
  │ emit({ type: "create_and_send", content: "..." })
  ▼
WsGateway.handleMessage(socket, envelope)
  │ → MessageBus.publish({ type: "create_and_send", clientId, payload })
  ▼
AiMessageRouter._handleCreateAndSend(clientId, content, context, emit)
  │
  │ 1. RoomService.create({ title: content.substring(0, 20) })
  │    └── room ← { id, title, status }
  │
  │ 2. emit({ type: "created", roomId: room.id })
  │
  │ 3. roomSession = RoomSession.create({ roomId, clientId, emit })
  │    └── holds: messages[], stateMachine (Idle), abortController
  │
  │ 4. Build callbacks bridge (all callbacks include roomId to match WorkflowCallbacks interface):
  │    callbacks = {
  │      onTextChunk: (rid, chunk) => roomSession.stateMachine.textChunk(chunk),
  │      onToolCall: (rid, info)    => roomSession.stateMachine.toolCall(info.toolCallId, ...),
  │      onLlmDone: (rid, reason)   => roomSession.stateMachine.llmDone(),
  │      onError: (rid, code, msg)  => roomSession.stateMachine.error(code, msg),
  │      onStop: (rid)              => roomSession.stateMachine.stop(),
  │    }
  │
  │ 5. RequestDispatcher.dispatch({ roomId, clientId, content, callbacks })
  │    │
  │    ├── rate limit check
  │    ├── Executor.execute({
  │    │     roomId, content, callbacks,
  │    │     abortSignal: roomSession.abortController.signal,
  │    │   })
  │    │     │
  │    │     ├── contextBuilder.build(roomId, tokenLimit)
  │    │     │     └── MessageService.buildLLMHistory(roomId, tokenLimit)
  │    │     │
  │    │     ├── callbacks.onContextReady(roomId)            // triggers BuildingContext → Processing
  │    │     │
  │    │     ├── MessageService.create({ roomId, role: "user", content })
  │    │     │
  │    │     ├── LLM call (stream)
  │    │     │     └── for each chunk: callbacks.onTextChunk(roomId, chunk)
  │    │     │
  │    │     ├── If tool calls in response:
  │    │     │     ├── callbacks.onToolCall(roomId, toolCallInfo)
  │    │     │     ├── ToolRouter.route(toolCalls)
  │    │     │     ├── ToolDispatcher.waitForResultsByRoom(roomId, timeout=30s)
  │    │     │     │     └── blocks until deliverResult() resolves or timeout
  │    │     │     ├── loop back to LLM call with tool results
  │    │     │     └── repeat until no more tool calls
  │    │     │
  │    │     ├── MessageService.create({ roomId, role: "assistant", content, toolCalls })
  │    │     │
  │    │     ├── callbacks.onLlmDone(roomId, finishReason)
  │    │     │
  │    │     └── return
  │    │
  │    └── finally: cleanup (executor instance discarded)
  │
  └── Client receives: created → text_chunk(s) → [tool_call] → done
```

## RoomStateMachine States & Transitions

```
                ┌─────────────────────────────────────────────────────┐
                │                                                     │
                │  Idle ──receiveMessage──▶ BuildingContext           │
                │                     │                               │
                │                     └──contextReady──▶ Processing   │
                │                                           │         │
                │                    ┌──────────────────────┤         │
                │                    │                      │         │
                │              ToolWaiting ◀──toolCall      │         │
                │                    │                      │         │
                │              ToolExecuting ◀──toolExecute │         │
                │                    │                      │         │
                │              Processing ◀──toolDone       │         │
                │                    │                      │         │
                │                    └──llmDone/            │         │
                │                       error/              │         │
                │                       stop ──────────────▶│         │
                │                                    Done   │         │
                │                                           │         │
                └───────────────────────────────────────────┘         │
```

Valid transitions matrix with **explicit event triggers**:

| From State | To State | Trigger Event |
|---|---|---|
| Idle | BuildingContext | `receiveMessage()` — user message received |
| BuildingContext | Processing | `contextReady()` — context built, about to call LLM |
| BuildingContext | Done | `error()` / `stop()` — failure or cancellation before LLM call |
| Processing | ToolWaiting | `toolCall(requiresConfirmation=true)` — needs user approval |
| Processing | ToolExecuting | `toolCall(requiresConfirmation=false)` — auto-execute |
| Processing | Done | `llmDone()` / `error()` / `stop()` |
| ToolWaiting | ToolExecuting | `toolExecute()` — user approved the tool call |
| ToolWaiting | Done | `error()` / `stop()` |
| ToolExecuting | Processing | `toolDone()` — all tool results collected, back to LLM |
| ToolExecuting | Done | `error()` / `stop()` |
| Done | (none) | Terminal state — no further transitions |

### RoomSession

RoomSession is the per-room state holder. It replaces RoomStateMachineFactory's role of tracking room-level state. Responsibilities are clearly delineated:

| Responsibility | Component | Description |
|---|---|---|
| **Protocol state** | RoomStateMachine | Idle → Processing → Done transitions |
| **Cancellation** | AbortController | Abort in-flight LLM calls |
| **Concurrency guard** | `isActive()` flag | Reject duplicate active sessions |
| **Lifecycle** | `create()` / `destroy()` | Map registration and cleanup |

RoomSession is managed by a `RoomSessionRegistry` (replaces `RoomStateMachineFactory`):

```typescript
class RoomSessionRegistry {
  private sessions = new Map<string, RoomSession>();
  private byClientId = new Map<string, Set<string>>();

  create(opts: { roomId: string; clientId: string; emit: EmitFn }): RoomSession {
    const existing = this.sessions.get(opts.roomId);
    if (existing && existing.isActive()) {
      throw new Error(`Room ${opts.roomId} already active`);
    }
    if (existing) this.destroy(opts.roomId);

    const session = new RoomSession(opts);
    this.sessions.set(opts.roomId, session);
    // ... register byClientId index
    return session;
  }

  get(roomId: string): RoomSession | null { /* ... */ }
  destroy(roomId: string): void { /* ... */ }
  destroyByClientId(clientId: string): void { /* ... */ }
}
```

```typescript
interface RoomSession {
  readonly roomId: string;
  readonly clientId: string;
  readonly stateMachine: RoomStateMachine;
  readonly abortController: AbortController;

  constructor(opts: { roomId: string; clientId: string; emit: EmitFn });

  abort(): void;
  isActive(): boolean;  // stateMachine.state !== Done
}
```

**Note on heartbeat timeout**: The current `AISessionManager` has a 2-minute heartbeat that aborts stale sessions. This will be replaced by a periodic cleanup in `RoomSessionRegistry` that scans for sessions in `Done` state older than a configurable threshold (e.g., 5 minutes) and removes them. Active sessions that have been idle for too long (e.g., client disconnected without proper disconnect event) will be aborted. This is a background task, not per-session heartbeat.

### Executor (per-execution instance)

The existing `WorkflowExecutor` class is refactored to become `Executor`. The core LLM loop logic (LangGraph stream, tool call handling) remains the same, but the class is no longer a NestJS singleton managing global state. Instead, each `dispatch()` call creates a new `Executor` instance (or uses a static factory that creates a fresh instance internally).

```typescript
interface ExecutionCtx {
  roomId: string;
  content: string;
  callbacks: WorkflowCallbacks;
  abortSignal: AbortSignal;
  llmConfigMap?: NodeLLMConfigMap;
  graphName?: string;
  tokenLimit?: number;
}

class Executor {
  constructor(private ctx: ExecutionCtx) {}

  async execute(): Promise<void> {
    // 1. Build context via ContextBuilder
    // 2. Optionally compress via ContextCompressor (future)
    // 3. Persist user message
    // 4. Call LLM (stream)
    // 5. Handle tool calls loop
    // 6. Persist assistant message
    // 7. Signal completion via callbacks
  }
}
```

Usage in RequestDispatcher:
```typescript
const executor = new Executor({
  roomId, content, callbacks,
  abortSignal: roomSession.abortController.signal,
  llmConfigMap: ctx.llmConfigMap,
});
await executor.execute();
// executor instance is discarded after this
```

## Approaches Considered

### Approach A: Instance-per-execution (chosen)
- **Summary**: 每次 AI 对话轮次创建独立 Executor 实例，执行完销毁。RoomSession 作为轻量状态持有者，在调用链中传递。
- **Effort**: L (5-8 天) — 涉及多个深层耦合模块的重构，建议分阶段实施
- **Risk**: Low
- **Pros**:
  - 执行单元天然隔离，无状态泄漏
  - 并发安全：每个请求独立实例，不需要并发控制
  - 内存管理简单：执行完即 GC，不需要后台清理
  - 符合"状态和代码完全无关"的设计目标
- **Cons**:
  - 每次执行需要重新构建上下文（可从缓存优化）
  - Room 级状态（如消息列表）需要通过 RoomSession 传递
- **Reuses**: 现有 LangGraph workflow, ToolDispatcher, MessageService — 只改变调用方式和依赖注入

### Approach B: Per-room Session Manager
- **Summary**: Room 对应一个长生命周期 Session 对象，复用处理多轮对话。
- **Effort**: M (4-6 天)
- **Risk**: Medium
- **Pros**:
  - 更符合"每个会话一个实例"的直觉
  - 上下文可以缓存，减少重复构建
- **Cons**:
  - 需要处理并发控制（用户快速发两条消息）
  - 需要超时清理和僵尸实例管理
  - 状态泄漏风险：前一轮的状态可能影响后一轮

## Recommended Approach

**Approach A** — Instance-per-execution。理由：
- 你之前遇到的 RoomStateMachineFactory 问题根源就是"用全局工厂管理实例生命周期"
- Approach A 彻底消除全局状态：Executor 实例随请求创建和销毁，RoomSession 在调用链中传递
- 并发安全、内存安全、状态隔离都不需要额外代码保证

## Open Questions

1. **Context caching**: 是否需要缓存已构建的上下文？多次消息之间系统 prompt 不变，只需增量更新消息历史。可以后续优化。
2. **Tool confirmation UX**: 前端需要用户确认的危险 tool call，当前流程中 ToolDispatcher.waitForResults() 已经支持等待前端返回结果，但 UI 交互流程需要确认。
3. **Error recovery**: 如果 Executor 在执行中崩溃（进程重启、网络中断），客户端重连后如何恢复？当前设计不处理——需要未来设计（可能涉及消息持久化 + 状态恢复）。

## Success Criteria

- 消除 RoomStateMachineFactory 单例依赖
- RoomOrchestrator 不再直接 import RoomStateMachineFactory
- 每个 Executor 实例独立创建和销毁，无全局状态
- 完整的测试覆盖：RoomSession、Executor、callbacks bridge
- 支持并发请求安全（同一 room 同时收到两条消息）
