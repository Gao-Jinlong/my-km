# Design: Multi-Agent Writing Studio — Agent Orchestrator via MessageBus

## Problem Statement

在知识管理工具 (my-km) 中，现有的 AI 对话架构 (LangGraph `chat-graph` + `RoomOrchestrator` + `MessageBus`) 支持单轮 LLM 调用循环。用户需要一种多 Agent 协作写作体验——主编、写手、研究员等角色各司其职，用户能实时看到每个 Agent 的工作状态并在任意环节介入修改。这类似于 Cursor/Copilot 的代码审查模式，但面向写作场景。

## What Makes This Cool

- **实时可视化多 Agent 工作流** — 用户能看到 "主编在分配任务"、"写手在起草第 2 段"、"研究员在查资料"，每个 Agent 的 state、thinking、output 实时推送
- **中途介入** — 用户在任何 Agent 完成其任务后、结果提交前，可以修改、批注、拒绝
- **自动化流水线** — 用户只给一个主题，Agent 团队自动完成从调研到成稿的全流程

## Constraints

- **复用现有基础设施** — `MessageBus` (apps/server/src/ws/message-bus.ts)、`RoomOrchestrator`、`GraphRegistry`、`LLMResolver`、`WorkflowExecutor`、WebSocket 通道
- **垂直场景 MVP** — 先跑通 "主编 Agent → 写手 Agent" 两个角色，验证 Agent 注册、任务分配、状态同步、用户介入链路
- **不重做 LLM 基础设施** — 每个 Agent 仍然通过 `LLMResolver` 获取 LLM 实例，可独立配置不同 LLM
- **LangGraph 图定义保持纯函数式** — 不引入 NestJS DI 到图定义中 (与现有 `chat-graph.ts` 一致)

## Premises

1. 现有 MessageBus 可以扩展为 Agent 间通信的骨干——只需新增 Agent 专属 message types，不需要新的通信基础设施
2. "主编 → 写手" 的垂直场景足够窄，能在几天内跑通核心链路
3. 用户介入的关键体验是 "看到 → 修改 → 继续"——Agent 状态必须通过 MessageBus 实时暴露给前端
4. 交付物是 apps/server/src/ai/ 下的新 Agent Orchestrator 模块 + 前端可视化 (后续实现)

## Approaches Considered

### Approach A: LangGraph 子图嵌套
- 利用 `StateGraph` 嵌套，主编父图 + 写手子图，状态自动暴露
- Effort: M | Risk: Low
- Pros: 复用 chat-graph 架构，子图天然状态同步
- Cons: LangGraph 子图调试差，state annotation 臃肿

### Approach B: 消息总线协调器 (推荐)
- 新建 `AgentOrchestrator` 服务，每个 Agent 是独立 `MessageHandler`，通过 `MessageBus` 传递任务和结果
- Effort: L | Risk: Med
- Pros: Agent 完全解耦，天然支持前端订阅任意 Agent 状态，更贴近 "真实团队" 隐喻
- Cons: 需要新建编排逻辑，状态一致性管理更复杂，初期投入比 Approach A 大一倍
- Reuses: `MessageBus`、WebSocket 通道、`LLMResolver`、现有 `Executor` 模式

### Approach C: 单图多节点平铺 (最小可用)
- 在一张 LangGraph 里定义多个平行 Agent 节点，条件边控制流转
- Effort: S | Risk: Low
- Pros: 最快 demo (1-2 天)，纯 graph 扩展
- Cons: 图复杂度随 Agent 增长失控，state schema 共享不灵活

## Recommended Approach

**Approach B: 消息总线协调器**

理由：用户选择了完全解耦的 Agent 架构，因为探索多 Agent 协作本身就是学习目标。你已有的 `MessageBus` 基础设施 (apps/server/src/ws/message-bus.ts) 为这个方案铺好了路——不需要新建通信层，只需要新增 Agent 专属的 message types 和 handler。

### 核心架构

**设计决策：AgentOrchestrator 是唯一的 MessageBus handler。** 所有 `agent:*` 入站消息都由 Orchestrator 处理；`AgentHandler` 是一个内部工具类，通过直接方法调用执行 Agent 的 LangGraph，不注册为 MessageHandler。这避免了 approve/reject 在 pub/sub 上的 race condition。

### Approve/Reject 状态流 (MVP: 不做 LangGraph checkpoint)

```
Orchestrator                          AgentHandler                     Frontend
    │                                     │                               │
    │ handle(agent:start)                 │                               │
    ├─ create session                     │                               │
    ├─ emit agent:status(assigned)        │                               │◀── status: assigned
    │                                     │                               │
    ├─ handler.execute(editorAgent) ─────▶│                               │
    │                                     │ executor.execute()            │
    │                                     │ ├─ callback.onTextChunk()     │◀── agent:thinking
    │                                     │ └─ callback.onComplete()      │◀── agent:output
    │                                     │                               │
    │◀── returns { output }               │                               │
    │                                     │                               │
    ├─ save session state                 │                               │
    ├─ emit agent:status(awaiting)        │                               │◀── status: awaiting_approval
    │                                     │     (handler 已返回，无阻塞)   │
    │                                     │                               │
    │         handle(agent:approve)       │                               │
    ├─ mark editor approved              │                               │◀── (internal state update)
    ├─ handler.execute(writerAgent) ─────▶│                               │
    │                                     │ executor.execute()            │
    │                                     │ └─ callback.onComplete()      │◀── agent:output
    │◀── returns { output }               │                               │
    ├─ emit agent:complete(finalDoc)      │                               │◀── complete
```

**关键设计：** `AgentExecutor.execute()` 是 async 方法，运行到完成即返回。Orchestrator 在两次 execute 调用之间保存 session 状态，等待用户的 approve/reject 消息。不需要 LangGraph checkpoint——agent 的运行状态是短暂的，决策状态（谁已完成、谁在等待）由 Orchestrator 的内存管理。如果用户 reject，Orchestrator 带着修改后的 prompt 重新调用 `handler.execute()`，相当于一次全新的运行。

### Message Types

```typescript
// Inbound (client → server → MessageBus)
// AgentOrchestrator 的 allowedTypes 精确列出这 4 个类型 (不支持通配符)
'agent:start'       // { sessionId?, topic, agentRoles: string[] }
'agent:approve'     // { sessionId }                批准当前等待的 agent 产出
'agent:reject'      // { sessionId, reason }         拒绝并要求当前 agent 重写
'agent:intervene'   // { sessionId, modification }   修改当前 agent 的 prompt 后重跑

// Outbound (server → client via socket.emit)
'agent:status'      // { sessionId, agentId, status, pipelineIndex }
'agent:thinking'    // { sessionId, agentId, chunk }  Agent 思考流 (streaming)
'agent:output'      // { sessionId, agentId, content } Agent 单次产出
'agent:complete'    // { sessionId, finalDocument }   全部 agent 完成
'agent:error'       // { sessionId, agentId, error }   Agent 出错
```

**注意：** 所有 inbound 消息的 payload 必须包含 `sessionId` 作为关联键。`AgentOrchestrator` 通过 `sessionId` 从 `AgentStateStore` 中查找对应的 session 上下文，决定哪个 agent 是该消息的目标。MVP 阶段 `sessionId` 放在 payload 中而非 `BusMessage` 的顶层字段（不修改 `BusMessage` 接口）。

### Agent Lifecycle (精确流程)

```
1. 客户端发 agent:start → WsGateway → MessageBus.publish()
   → AgentOrchestrator.handle() 被调用 (allowedTypes 精确匹配 'agent:start')

2. Orchestrator 创建 AgentSession:
   - 生成 sessionId
   - 从 AgentRegistry 按 agentRoles 查找 AgentDefinition，按 pipelineStage 排序
   - 初始化每个 agent 的状态为 PENDING
   - 保存到 AgentStateStore (Map<sessionId, AgentSession>)
   - emit agent:status → 前端显示 pipeline 概览

3. Orchestrator 调用 pipeline 中第一个 agent:
   - 设置该 agent 状态为 ASSIGNED → RUNNING
   - 构造 callbacks: { onTextChunk, onComplete, onError }
     - onTextChunk → emit agent:thinking (通过 WsGateway.socket.emit)
     - onComplete → emit agent:output
     - onError   → emit agent:error
   - 直接调用 AgentHandler.execute(agentDef, input, callbacks)
   - AgentHandler 创建 AgentExecutor → AgentExecutor.execute() → async 运行到完成
   - 期间通过 callbacks 实时推送 thinking chunks 和 output
   - execute() 返回后，emit agent:output (最终产出)

4. Agent 完成后，Orchestrator 设置状态为 AWAITING_APPROVAL
   - emit agent:status(status: 'awaiting_approval')
   - AgentHandler.execute() 已返回，无阻塞
   - Orchestrator 等待下一个 inbound 消息

5. 用户发 agent:approve → MessageBus → Orchestrator.handle():
   - 查找 sessionId，标记当前 agent 为 APPROVED，产出追加到 document
   - 推进到下一个 agent，重复步骤 3

   用户发 agent:reject → MessageBus → Orchestrator.handle():
   - 标记当前 agent 为 REJECTED
   - 带着 reject reason 追加到 systemPrompt，重新调用 AgentHandler.execute()

   用户发 agent:intervene → MessageBus → Orchestrator.handle():
   - 带着用户修改指令重写 input prompt，重新调用 AgentHandler.execute()

6. 最后一个 agent APPROVED → emit agent:complete({ sessionId, finalDocument })
```

**调度机制总结：**
- Orchestrator ↔ Handler 是**直接方法调用**，不是 MessageBus
- 所有客户端消息走 **MessageBus**，Orchestrator 是唯一消费者
- approve/reject 之间**不存在 race condition**——Orchestrator 是同步状态机，每次 handle() 调用处理一个事件，更新状态，然后返回

### 接口定义

```typescript
// === agent.types.ts ===

/** Agent 角色定义，注册到 AgentRegistry */
interface AgentDefinition {
    role: string;                        // 'editor' | 'writer' | 'reviewer'
    systemPrompt: string;                // MVP: 静态字符串
    llmConfig?: LLMConfig;               // 可选: 覆盖默认 LLM
    pipelineStage: number;               // 执行顺序 (1 = 最先)
    requiresApproval: boolean;           // true = 等待用户批准后再推进
    maxRetries?: number;                 // 被 reject 后的最大重试次数 (默认 3)
}

/** 单个 agent 的运行时状态 */
interface AgentState {
    agentId: string;                     // 自动生成: `${sessionId}--${role}`
    role: string;
    status: 'pending' | 'assigned' | 'running' | 'awaiting_approval'
         | 'approved' | 'rejected' | 'error' | 'cancelled';
    output?: string;                     // 当前 agent 的产出
    retries: number;                     // 已重试次数
    startedAt?: Date;
    completedAt?: Date;
}

/** 一次完整的写作 session */
interface AgentSession {
    sessionId: string;
    clientId: string;                    // 发起请求的用户
    topic: string;                       // 写作主题
    agents: AgentState[];                // 按 pipelineStage 排序
    document: string;                    // 累积的文档内容 (每个 agent approved 后追加)
    status: 'running' | 'complete' | 'error' | 'cancelled';
    currentAgentIndex: number;           // 当前正在执行的 agent 索引
    createdAt: Date;
    abortController: AbortController;    // 支持用户取消整个 pipeline
}

/** Orchestrator dispatch callbacks (由 WsGateway 提供 socket 引用) */
interface AgentCallbacks {
    onThinking(sessionId: string, agentId: string, chunk: string): void;
    onOutput(sessionId: string, agentId: string, content: string): void;
    onError(sessionId: string, agentId: string, error: string): void;
    onStatus(sessionId: string, agentId: string, status: string): void;
}
```

### AgentHandler 与 Executor 的关系

**关键决策：不直接复用 `Executor`，新建轻量 `AgentExecutor`。**

现有 `Executor` 深度耦合了对话流基础设施：
- `messageService.buildLLMHistory(roomId)` — 从 DB 加载历史（Agent 不需要）
- `messageService.create(...)` — 持久化消息到 DB（Agent 不需要）
- `toolDispatcher.waitForResultsByRoom(...)` — 等待前端返回工具结果（Agent 不需要）
- `toolRouter.route(...)` — 路由工具调用到前端（Agent 不需要）

`AgentExecutor` 复用 Executor 的核心模式 (graph stream 循环 + abort check + tool call handling)，但去掉 DB 和前端依赖：

```typescript
// apps/server/src/ai/agents/agent-executor.ts

interface AgentExecutorCtx {
    sessionId: string;
    agentId: string;
    input: string;                       // 当前 agent 的输入 prompt
    callbacks: AgentCallbacks;
    abortSignal: AbortSignal;
    llmConfig?: LLMConfig;               // 直接指定 LLM，不用 NodeLLMConfigMap
    graphName?: string;                  // 默认 'chat'
}

export class AgentExecutor {
    // 与 Executor 相同的依赖 (纯服务，不依赖 NestJS DI)
    constructor(
        private ctx: AgentExecutorCtx,
        private deps: {
            graphRegistry: GraphRegistry;
            llmResolver: LLMResolver;
            toolDispatcher: ToolDispatcher;  // 仅用于 getDefinitions() 获取工具列表
            toolRouter: ToolRouter;          // 仅用于 needsConfirmation()
        },
    ) {}

    async execute(): Promise<{ output: string }> {
        // 1. 获取图定义 → 编译 (复用 Executor.getOrCreateGraph 模式)
        // 2. 创建 LLM caller (复用 Executor.createLLMCaller 模式，用 Agent 的 llmConfig)
        // 3. 获取工具列表 (toolDispatcher.getDefinitions())
        // 4. 构建初始 state: { messages: [{ role: 'user', content: input }] }
        // 5. 循环 graph.stream():
        //    - 每个 chunk → callbacks.onThinking()
        //    - abort check
        //    - 如果有 tool calls → 在 server 侧同步执行 (不等待前端)
        //      * 因为 Agent 的工具是 server 侧实现的，不是前端工具
        // 6. 返回最终 output
        // 7. callbacks.onOutput(output)
    }
}
```

**注意：** Agent 的工具调用模式与对话不同。对话的工具是前端实现的（如搜索、代码执行），需要等待前端返回结果。Agent 的工具是 server 侧纯函数（如字数统计、格式校验），同步执行即可。MVP 阶段 Agent 不需要工具，这个路径先留空。

### State Machine per Agent

```
PENDING → ASSIGNED → RUNNING → OUTPUT_READY → AWAITING_APPROVAL
    ↓         ↓         ↓           ↓                    ↓
  ERROR   CANCELLED  ERROR     user:intervene      APPROVED → 推进到下一个 agent
                               (重新输入 prompt)       ↓
                               ↘ RE-RUNNING      REJECTED → 重新运行同一 agent
                                     ↓                    ↓
                                     RUNNING ←────────────┘

 intervene 只允许从 AWAITING_APPROVAL 状态触发
 reject 后有 maxRetries 限制，超过后标记为 ERROR 并终止 pipeline
 agent:approve 后如果无下一个 agent → 标记 session 为 COMPLETE
```

### 错误恢复与超时

- **LLM 超时**: AgentHandler 内部复用 Executor 已有的 30s tool-result timeout。超时 → `onError` callback → `agent:error` → session 标记为 ERROR
- **Agent 崩溃**: `MessageBus.publish` 的 try/catch 会捕获 handler 异常并日志。Orchestrator 需要在 session 级别设置 `abortController.signal`，超时后主动 abort
- **Server 重启**: MVP 阶段不做 session 持久化。Server 重启后所有 in-flight session 丢失，前端检测到 connection lost 后提示用户重新开始
- **Token/Cost 限制**: MVP 阶段不设硬限制，但每个 AgentDefinition 有 `maxRetries` (默认 3) 防止无限循环

## Open Questions

1. **用户介入修改时，prompt 重组策略** — reject 时把 reject reason 追加到 systemPrompt 作为 "上次的产出被拒绝，原因是 X，请重新生成"；intervene 时把用户修改指令作为新的 user message。这比重建完整上下文更节省 token，但对多轮迭代的效果需要实验验证
2. **并行 Agent 支持** — MVP 阶段不支持同 stage 多 Agent 并行。如果未来需要，可以在 Orchestrator 中对相同 pipelineStage 的 agent 并发调用 `handler.execute()`，通过 Promise.all 等待全部完成

## Success Criteria

- 用户在终端/前端输入一个写作主题，Editor → Writer 自动协作，用户能看到两个 Agent 的状态流转
- 用户能在 Writer 产出后介入修改（"这段太长了，精简到 100 字以内"），Writer 重新生成
- 整个链路通过现有的 WebSocket + MessageBus 推送状态到前端，不需要新建通信通道
- 新增一个 Agent 角色（如 Reviewer）只需要在 registry 注册一个 `AgentDefinition`，不需要改 orchestrator 核心逻辑
