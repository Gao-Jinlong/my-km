# Checkpointer & Run 状态持久化方案

Generated by /plan-eng-review on 2026-06-07
Branch: main
Commit: ce3c558

## 问题分析

### Thread 消息未持久化的根因

1. `CHECKPOINTER_BACKEND` 环境变量未设置，默认为 `memory`（`MemorySaver`），进程重启即丢失
2. 即使切到 PostgresSaver，checkpointer 存的是 graph 内部 checkpoint，不是前端直接消费的消息格式
3. `executeRunProtocol` 只通过 SSE 发送消息，从未写入数据库的 Message 表
4. `MessageService` 只有查询方法，没有写入方法
5. `getThreadState` 从 `MessageService.findByRoomId` 读消息 → 永远返回空

### Run 状态未持久化的根因

1. `RunManager` 使用 `Map<string, RunRecord>` 存储运行中的 Run（纯内存）
2. Prisma Run 表已定义（status/token/timestamps），但 RunManager 从未调用 prisma
3. RunRecord（进程内对象）和 Run（Prisma 模型）是两个完全独立的体系

## 决策记录

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| D1 | Checkpointer 后端 | PostgresSaver | graph 内部状态完整持久化 |
| D2 | Run 状态持久化 | RunManager 同步写 DB | 最小改动，Run 表已就绪 |
| D3 | 消息提取方式 | CheckpointReaderService | 从 checkpoint 提取消息 |
| D4 | Run 查询 API | 新建 RunsController | 职责分离，符合 LangGraph 协议 |
| D5 | Message 表 | 删除，纯 checkpoint | 单一数据源 |
| D6 | Thread.messageCount | 删除 | 消息数从 checkpoint 动态计算 |
| D7 | Run ID 生成 | UUID | 跨 thread 全局唯一 |
| D8 | 测试覆盖 | 完整单测 | 核心路径必须有测试 |
