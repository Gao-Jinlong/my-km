## Context

当前工作空间 UI 已实现基本布局结构，但与设计稿 (docs/designs/my-km.pen) 对比存在以下差异：

1. **设计稿结构**: 包含深色 (Workspace) 和浅色 (Workspace Light) 两个完整主题的设计
2. **当前实现**: 基本布局完成，但细节样式和主题对齐不足
3. **约束**: 需保持现有组件结构，主要调整样式和视觉细节

## Goals / Non-Goals

**Goals:**
- 对照设计稿完成 UI 样式对齐
- 确保深色/浅色主题颜色正确应用
- 优化组件间距、图标尺寸等视觉细节
- 保持现有代码结构和组件层级

**Non-Goals:**
- 不改变现有组件架构
- 不添加新的功能特性
- 不修改工作空间状态管理逻辑

## Decisions

### 1. 主题颜色映射

**决策**: 使用现有的 CSS 变量系统 (`--ws-*`) 映射设计稿颜色

**理由**:
- 项目已建立 workspace 主题变量系统
- 便于主题切换和维护
- 符合 shadcn/ui 的设计模式

**替代方案考虑**:
- 直接使用设计稿的十六进制颜色 → 不利于主题切换
- 使用 Tailwind 默认颜色 → 无法精确匹配设计稿

### 2. 图标尺寸标准化

**决策**: Activity Bar 图标使用设计稿规格 (40x40 容器 + 20px 图标)

**理由**:
- 设计稿明确定义了尺寸规范
- 40x40 提供更好的点击区域
- 圆角背景提供清晰的选中状态

### 3. 组件样式实现方式

**决策**: 使用 Tailwind CSS 类 + 内联样式组合

**理由**:
- 保持与现有代码风格一致
- Tailwind 处理响应式和状态
- 内联样式处理精确尺寸

### 4. 视觉对比工作流

**决策**: 使用 Pencil MCP + Playwright/Chrome DevTools 进行截图对比

**理由**:
- Pencil MCP 可以直接从设计稿 (.pen 文件) 生成截图
- Playwright/Chrome DevTools 可以捕获浏览器中的实际渲染效果
- 并排对比可以快速识别视觉差异
- 可以在修复后重新截图验证

**工作流程**:
1. 使用 `pencil.get_screenshot` 获取设计稿组件截图
2. 使用 `playwright` 或 `chrome-devtools` 获取实现截图
3. 并排对比，记录差异
4. 修复后重新截图验证

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 主题颜色变量与设计稿不完全匹配 | 逐一对比设计稿变量，更新颜色值 |
| 组件间距调整影响布局 | 在开发环境测试多种屏幕尺寸 |
| 图标尺寸变化影响视觉平衡 | 参照设计稿比例，进行视觉验收 |

## Migration Plan

1. 更新主题颜色变量
2. 逐个组件对照设计稿调整样式
3. 在开发环境验证深色/浅色主题
4. 进行视觉回归测试
